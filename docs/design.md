# 0. サマリー

## 0.1. 本環境について

本リポジトリは、Arch Linux をベースとした個人用の開発環境である。ウィンドウマネージャは Sway と Hyprland の両方に対応している。

単なる設定ファイル（dotfiles）の管理ではなく、

* 環境を理解しながら運用できること
* OSやソフトウェアの仕組みを学習できること
* 環境を短時間で再構築できること

を目的として設計・運用している。

設定ファイルだけでなく、環境構築手順や運用方針もGitで管理し、再現性と保守性を高めている。

本環境を適用した結果、CPU使用率やメモリ使用量を削減でき、起動時間が短縮した。また、キーボード中心で一貫した操作体系を実現できた。

詳細な測定結果は第5章で示す。

---

## 0.2. 設計方針

本環境では、次の4つを設計上の基本方針としている。

* **理解・管理できる範囲を広げること**
* **思考と作業の速度の差を減らすこと**
* **操作体系を統一すること**
* **環境を再現しやすく保守しやすくすること**

これらを実現するために、

* Arch Linux
* Sway / Hyprland
* Gitによるdotfiles管理
* Snapperによるスナップショット管理
* CLI・TUI中心のソフトウェア

を採用している。

# 1. 設計目標

## 1.1. 背景

### 1.1.1. 古いPCでも快適に利用したい

現在利用しているPC（MouseComputer MB-J350）は比較的古い世代のノートPCであり、Windows環境ではCPU負荷が高く、ファンの動作音も大きくなりやすい。

そのため、本環境ではCPU使用率やメモリ使用量を抑え、限られたハードウェア資源を効率的に利用することを重視する。

### 1.1.2. 環境を短時間で復旧できるようにしたい

現在はPCを1台しか所有していない。

そのため、アップデートや設定変更によって問題が発生した場合でも、短時間で以前の作業環境へ復旧できることを重視する。

### 1.1.3. Linuxを学習対象として活用したい

本環境は、日常的な作業環境であると同時に、Linuxやソフトウェアの仕組みを学習するための環境でもある。

単に利用するだけではなく、「なぜその構成になっているのか」を理解しながら運用することを目的としている。

---

## 1.2. 設計方針

本環境は、前節の背景を踏まえ、次の4つを設計上の基本方針とする。

### 1.2.1. 理解・管理できる範囲を広げる

導入するソフトウェアや設定について、「何が動作しているのか」「なぜ必要なのか」を把握できる状態を目指す。

### 1.2.2. 思考と作業の速度の差を減らす

頭の中で考えた操作を、できるだけ少ない手数・待ち時間で実行できる環境を目指す。

### 1.2.3. 操作体系を統一する

可能な限りキーボード中心の操作体系を採用し、入力方法や操作方法を頻繁に切り替えなくても作業できる環境を目指す。

### 1.2.4. 環境を再現しやすく保守しやすくする

OSの再インストールや新しいPCへの移行時にも、短時間で同じ環境を再構築できることを目指す。

---

## 1.3. 評価指標

本環境では、設計が目的を達成できているかを、次の指標で評価する。

| 指標                 | 評価基準    |
| ------------------ | ------- |
| CPU使用率             | 低いほどよい  |
| メモリ使用量             | 少ないほどよい |
| 起動時間               | 短いほどよい  |
| マウス操作回数            | 少ないほどよい |

# 2. 設計の具体的実現

本章では、第1章で示した設計方針をどのような技術やソフトウェアによって実現しているかを説明する。

---

## 2.1. OS

### 2.1.1. Linuxを採用する理由

WindowsではなくLinuxを採用した理由は、次の3点である。

* オープンソースであり、OSの仕組みまで理解しやすいため、学習環境として適している。（1.1.3、1.2.1）
* 不要な常駐ソフトウェアやサービスが少なく、古いPCでも軽快に動作する。（1.1.1）
* 更新のタイミングを自分で管理できるため、作業を中断されにくく、環境の保守もしやすい。（1.1.2）

---

### 2.1.2. Arch Linuxを採用する理由

#### 最小構成から構築できる

必要なソフトウェアだけを自分で選択して導入できるため、環境全体を把握・管理しやすい。

また、不要なサービスを導入しないことで、CPU使用率やメモリ使用量の削減にもつながる。（1.2.1、1.2.4）

#### ローリングリリース

一度に大規模なOSアップグレードを行う必要がなく、小さな更新を継続的に適用できる。

更新内容を確認しながら保守できるため、環境の変化を把握しやすい。（1.1.2）

#### ArchWiki

ArchWikiはLinux全般を含めた情報量が非常に多く、OSやソフトウェアの仕組みを学ぶ上で有用である。（1.1.3）

#### パッケージ管理

ソフトウェアは、可能な限り公式のパッケージマネージャから導入する。

優先順位は次のとおりである。

1. pacman（公式リポジトリ）
2. Cargo
3. npm
4. AUR

この方針により、導入・更新・削除の方法をできるだけ統一できる。

また、AURを利用する場合でもPKGBUILDを確認できるため、第三者のインストールスクリプトを直接実行する場合と比較して、実行内容を把握しやすい。（1.2.1）

---

## 2.2. ウィンドウマネージャ

### 2.2.1. タイリングWMを採用する理由

キーボードのみでウィンドウ操作を完結できるため、ブラウザやGUIアプリケーションも含めて一貫した操作体系を実現できる。

tmuxやZellijでは管理できないGUIアプリケーションまで同じ思想で扱えることも利点である。（1.2.3）

### 2.2.2. Swayを採用する理由

Swayは次の設計方針に適している。

* 軽量であり、古いPCでも快適に動作する。（1.1.1）
* アニメーションが少なく応答性が高いため、思考から操作までの時間を短縮できる。（1.2.2）
* Waylandベースであるため、将来的なLinux環境への移行時にも設定を再利用しやすい。（1.2.4）

### 2.2.3. Hyprland にも対応する理由

当初は Sway のみを採用していたが、現在は Hyprland にも対応し、どちらの WM でも同じワークフロー（キーバインド・Waybar・通知・電源メニュー）を再現できる構成としている。

* Hyprland は近年人気が高く利用者が多いため、Hyprland を使いたい場合でも本リポジトリの構成をそのまま利用できるようにした。
* タイリングWMとしての有用性（2.2.1）は、どちらの WM でも変わらない。
* マシン負荷にも大きな差はない。（同条件での実測値は5.3を参照）
* 使い分けの目安として、i3 互換の安定した最小構成を求める場合は Sway、アニメーションや装飾などのカスタマイズ性を重視する場合は Hyprland が適している。

---

## 2.3. ソフトウェア選定方針

### 2.3.1. キーボード中心で操作できること

CLI・TUIを優先し、GUIであってもキーボード操作を前提としたソフトウェアを採用する。

例：Yazi、Zathura、btop、Tridactyl

---

### 2.3.2. 思考から操作までの時間を短縮すること

操作回数や待ち時間を減らせるソフトウェアを優先する。

例：fzf、zoxide、btop

---

### 2.3.3. 操作体系を統一すること

キーボード中心の操作体系を維持するため、入力方法やキーバインドも可能な限り統一する。

例：xremap

---

### 2.3.4. 復旧しやすい環境を構築すること

OSの再インストールやトラブル発生時にも短時間で元の環境へ戻せる構成を目指す。

例：Snapper、dotfiles、install-common.sh / install-hypr.sh / install-sway.sh

---

## 2.4. 例外

基本方針はCLI・TUI中心であるが、用途によってはGUIアプリケーションを採用する。

### 2.4.1. GUIが適している用途

動画・画像・音声編集など、GUIの方が効率的な用途ではGUIアプリケーションを利用する。

例：VLC、Drawing

### 2.4.2. サービス利用を優先する用途

音楽・ゲーム・SNSなど、代替アプリケーションを利用することよりもサービス自体を利用する価値が高い場合は、公式クライアントを利用する。

例：Steam、Spotify、Discord

# 3. dotfilesをGitで管理する理由

本リポジトリでは、設定ファイルだけでなく、環境構築手順もGitで管理している。

目的は、環境の変更履歴を残すことだけではない。

OSの再インストールや新しいPCへの移行、設定変更によるトラブル発生時にも、短時間で同じ環境を再現できることを重視している。

## 3.1. 環境の再現性を高める

OSを再インストールした場合でも、このリポジトリを取得して環境構築スクリプトを実行するだけで、普段利用している開発環境を短時間で復元できる。

また、新しいPCへ移行する場合も、設定を一から作り直す必要がない。

## 3.2. 変更履歴を管理する

設定ファイルをGitで管理することで、

* 変更履歴を残せる
* 問題が発生した場合に以前の状態へ戻せる
* 変更内容を比較・レビューしやすい

といった利点が得られる。

## 3.3. 保守対象を明確にする

Gitで管理するのは、環境を構成する設定ファイルやスクリプトのみとする。

プロジェクトや個人データは管理対象に含めず、それぞれ独立して管理する。

これにより、環境とデータを分離し、保守性と移行性を高める。

---

# 4. ソフトウェア導入・保守方針

本環境では、「導入しやすさ」よりも「理解・保守しやすさ」を優先する。

## 4.1. パッケージマネージャを優先する

ソフトウェアは、可能な限り公式に提供されているパッケージマネージャから導入する。

優先順位は次のとおりである。

1. pacman
2. Cargo
3. npm
4. AUR

これにより、導入・更新・削除方法を統一し、環境全体を管理しやすくする。

## 4.2. AUR利用時の確認事項

AURを利用する場合は、PKGBUILDを確認し、実行内容を把握した上で導入する。

必要に応じて、以下についても確認する。

* GitHubリポジトリ
* メンテナンス状況
* 最終更新日
* Contributors
* Issues
* Releaseの有無
* 署名・ハッシュによる改ざん防止

外部のインストールスクリプトを安易に実行するのではなく、可能な限り導入方法を理解した上で利用する。

## 4.3. システムアップデート

部分アップグレードは避け、`pacman -Syu` による更新を基本とする。

## 4.4. 更新前のバックアップ

システムに大きな変更を加える前や、不安のあるアップデートを適用する前には、必要に応じて Snapper でスナップショットを作成する。

## 4.5. 保守方針

環境を継続して利用するため、以下の方針で保守を行う。

* 設定ファイルはGitで管理する
* プロジェクトや個人データはGit管理対象外とする
* 環境とデータを分離して管理する
* ソフトウェアは定期的に更新する
* 環境を変更した場合は、必要に応じて本リポジトリへ反映する

---

# 5. 設計結果

本章では、第1章で示した評価指標に対する結果を示す。

本章の数値は、信頼度の異なる2種類に分けて記載する。

* **測定値**：`scripts/benchmark.sh` および `scripts/count-mouse` により
  再現可能な手順で測定した値
* **参考値**：過去の環境（Ubuntu）で btop の表示を目視で読み取った値。
  測定手順が統一されておらず再現できないため、傾向の参考に留める

なお、現環境をスクリプトで実測したところ、以前に目視で読み取っていた
値（CPU使用率 約10%）に対して実測は 2.7% であり、目視測定が大きく
過大であったことが判明した。この経験から、本章の数値は測定方法を
明記した上で、再現可能な測定値を基準とする方針に改めた。

---

## 5.1. 測定環境

* PC：MouseComputer MB-J350
* CPU：Intel Core i5-8250U
* メモリ：8GB DDR4
* GPU：Intel UHD Graphics 620
* ストレージ：SATA SSD

---

## 5.2. 測定方法

CPU使用率・メモリ使用量は `scripts/benchmark.sh` により測定した。
`/proc/stat` の差分からCPU使用率を、`free` からメモリ使用率（used / total）を
算出し、10秒間隔で5回サンプリングした平均値を記載している。

起動時間は `systemd-analyze` の出力による。

マウス操作回数は、`scripts/count-mouse` により libinput の入力イベントを
10分間計数した値である。クリックはボタン押下イベントの回数、移動は
1秒以上の停止で区切った連続移動を1回として数え、両者の合計を
「操作回数」とする。なお、5.4 の Ubuntu 環境の参考値は手動計数に
よるもので、定義が同一でない点に留意されたい。

測定は、起動から5分以上経過後、Firefoxとターミナルを起動した
通常利用状態で実施した。

---

## 5.3. 現環境の測定値（Arch + Sway / Hyprland）

測定日：Sway 2026-07-08 / Hyprland 2026-07-10

いずれも 5.2 と同一の手順（起動から5分以上経過後、Firefoxとターミナルを起動した通常利用状態）で測定した。

| 項目                       |                Sway |            Hyprland |
| ------------------------ | ------------------: | ------------------: |
| CPU使用率                   | 2.7%（5回平均、2.1〜3.4%） | 4.2%（5回平均、2.5〜6.0%） |
| メモリ使用量                   | 28.2%（約2.2GB / 8GB） | 25.7%（約2.1GB / 8GB） |
| 起動時間（電源投入〜graphical.target） |              11.5秒 |              10.4秒 |
| マウス操作回数（10分あたり）          |   7回（クリック2回・移動5回） |                未測定 |

起動時間の内訳（systemd-analyze）：

| 段階        | Sway | Hyprland | 内容                  |
| --------- | ---: | -------: | ------------------- |
| firmware  | 3.3秒 |     3.4秒 | UEFI初期化（OS外）        |
| loader    | 1.0秒 |     0.7秒 | ブートローダ              |
| kernel    | 1.4秒 |     1.3秒 | カーネル初期化             |
| initrd    | 2.0秒 |     2.0秒 | 初期RAMディスク           |
| userspace | 4.0秒 |     3.0秒 | systemdによるサービス起動    |

OSが関与する範囲（kernel以降）は、Sway 約7.4秒、Hyprland 約6.3秒である。

両WMの測定値の差については、次のように解釈している。

* CPU使用率は Hyprland がやや高いが、サンプル範囲（2.1〜3.4% と 2.5〜6.0%）が重なっており、5回サンプリング1回分の測定では WM 間の有意な差とは判断できない。
* 起動時間（graphical.target 到達）はログインマネージャ起動までの時間であり、WM 本体の起動は含まれない。差は WM の違いではなく測定時のばらつきによるものと考えられる。
* 総じて、マシン負荷に大きな差はないという判断（2.2.3）と整合する結果である。

## 5.5. 評価指標の見直し

当初は「ログイン後に操作可能になるまでの時間」も評価指標としていたが、
測定を検討した結果、この時間はパスワード入力などの人間側の操作時間が
支配的であり、環境の設計判断を評価する指標として適さないと判断した。

また、この指標が本来捉えようとしていたセッションの軽さは、
CPU使用率・メモリ使用量・起動時間の3指標で既に表現されている。

以上の理由から、本指標は評価対象から除外した。

---

## 5.6. まとめ

参考値との比較では、CPU使用率・メモリ使用量ともに削減の傾向が確認できた。
また、Sway と Hyprland を同一手順で測定した結果、両WM間で負荷に大きな差がないことを確認できた。

加えて、測定手順をスクリプトとして残したことで、今後の設定変更や
ハードウェア移行時にも同一条件で比較できるようになった。

目視測定と実測の乖離や、評価指標の見直しを経験したことも含め、
「計測は再現可能な手順と指標の妥当性の検討とセットで初めて意味を持つ」
という運用上の教訓を得た。
